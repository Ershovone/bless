# /take-task — один цикл «возьми задачу и сделай»

Ты — оркестратор. Сам ты код не пишешь, ты только координируешь.

## Шаги

### 1. Проверка предусловий
- `git status` — рабочая директория должна быть чистой
- `git branch --show-current` — должна быть `main`
- Если нет — выведи `SKIP: dirty working tree` и остановись (не пытайся ничего чинить — это пользовательские in-progress изменения).

### 2. Найти задачу
Просканируй `.claude/tasks/*.md` (исключая `README.md`). Распарси frontmatter каждого. Возьми **одну** с `status: open`, наименьшим `priority`, при равенстве — лексикографически первое имя файла.

Если ни одной open-задачи нет — выведи `IDLE: no open tasks` и остановись.

### 3. Зарезервировать
Через Edit обнови frontmatter выбранной задачи: `status: in_progress`, `attempts: <текущее+1>`. Это исключит race-condition при параллельных вызовах.

### 4. Спавн worker-а
Вызови Agent с `subagent_type: "bless-worker"`. Промпт:

```
Возьми задачу из файла: <абсолютный путь>

Прочитай задачу целиком, следуй своему системному протоколу. Финальный ответ — одна строка RESULT: ...
```

Запускай в foreground (нужен результат для следующего шага).

### 5. Обработать результат
Парси последнюю строку ответа агента:
- `RESULT: done <url>` → ничего не делаем (worker сам обновил frontmatter и сделал PR). Выведи `OK: done <url>`.
- `RESULT: failed <reason>` → worker сам уже вернул задачу в open и записал заметку. Выведи `RETRY: <id> — <reason>`.
- `RESULT: skipped <reason>` → откат: верни `status: open`, **декрементируй** attempts обратно (это не была настоящая попытка). Выведи `SKIP: <id> — <reason>`.
- Если строки `RESULT:` нет вообще (агент завис/сломался) → откат как при skipped + добавь заметку в задачу `## Заметка: оркестратор не получил RESULT`. Выведи `ERROR: no result from worker`.

### 6. Финиш
Не запускай следующую итерацию сам. Если пользователь дёрнул через `/loop`, оркестратор-loop вызовет тебя заново. Если просто `/take-task` — один цикл и стоп.

## Важно
- Один вызов = одна задача. Никогда не бери две за раз.
- Никаких `git push` в этом скилле — это делает worker.
- Если worker создал ветку, но упал до PR и не убрал её — оркестратор НЕ удаляет ветки. Просто оставляет как есть и репортит пользователю.
